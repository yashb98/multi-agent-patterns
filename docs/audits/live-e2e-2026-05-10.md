## Session-introduced fixes

### Fix A — `jobpulse/form_engine/file_uploader.py:upload_pdf`

**Why:** Greenhouse's React file widget destroys `<input id="resume">` after `set_input_files` succeeds, replacing it with an "uploaded" UI state element. The post-upload `_readback_files_length(locator)` then times out resolving the (now-gone) locator → returns -1 → triggers retry → retry's `locator.set_input_files(...)` times out 30 s → `FileUploadError` raised, even though the CV had been uploaded.

**Verified live (2026-05-10 run1.2):** `page_has_filled_resume: True`; `Yash_Bishnoi_Anthropic.pdf` visible in page DOM; original input element absent.

**How:** When readback returns -1, fall back to `page.body.innerText.includes(filename)` check before raising. If filename is present, treat upload as successful.

**Confirmed working (run4):**
```
upload_pdf: ✓ uploaded Yash_Bishnoi_Anthropic.pdf — input was re-rendered post-upload, filename visible in page DOM
upload_pdf: ✓ uploaded Cover_Letter_Anthropic.pdf — input was re-rendered post-upload, filename visible in page DOM
```

### Fix B — `jobpulse/native_form_filler.py:_resolve_listbox_scope`

**Why:** Item 1's listbox scoping for visa Yes/No was supposed to prevent the global `[role='option']` selector from picking 'Norway' from an open phone-country picker. The fix landed in `e031f60` but the scoping XPath `ancestor::*[contains(@class,'select__control')][1]/following-sibling::*[contains(@class,'select__menu')][1]` returns `count=0` on Anthropic's React Select v5+ structure — the menu is NOT a sibling of `.select__control`; it is mounted inside `.select-shell` as a separate child once the menu opens.

**Verified live:** XPath returned 0 matches. Verified counts:
- predictable id: `react-select-question_4089394008-listbox` → count=1, options=['Yes','No']
- `.select-shell` descendant `.select__menu` → count=2 (menu + menu-list)

**How:** Two new fallback layers ahead of the legacy XPath: (1) predictable React Select listbox id; (2) `.select-shell` descendant.

### Fix C — `jobpulse/form_engine/field_resolver.py:canonicalize_country_value`

**Why (real root cause of the 'Norway' bug):** With Fix B in place, the visa fill was STILL writing `'Norway'`. Tracing showed `fill_value` was `'Norway'` BEFORE the option lookup. Root cause: `canonicalize_country_value` triggered on any label containing `"country"` (substring match), and the visa label contains it: *"…require employment visa sponsorship to work in the **country** in which the job…"*. With value `"No"` (the screening cache answer), the function looked up `"no"` in `_COUNTRY_DATA` → matched Norway's ISO 3166-1 alpha-2 code → returned `"Norway"`.

**How:** Bail out of canonicalization when:
- the value is an unambiguous Yes/No (`yes|no|y|n|true|false`); OR
- the label contains visa/work-auth keywords (`visa`, `sponsorship`, `right to work`, `authorized to work`, `will you require`).

**Sanity check (advisor concern):** `grep -rE "(right to work|authorized to work).*country|country.*right to work|country.*authorized to work" tests/fixtures/ tests/jobpulse/` returned **zero matches**. No fixture has a label like "Country (right to work)" that would be mis-handled by the bail.

**Confirmed working (run4):**
```
fill ✓ 'Will you now or will you in the future require employment vi' = 'No' [tech=combobox_type_to_search, expected='No']
```

### Fix D — `jobpulse/page_analysis/page_reasoner.py:_parse_response`

**Why:** Run3 aborted at the page reasoner: Kimi returned JSON that `json.loads` rejected (`Expecting ',' delimiter: line 19 column 197`). The reasoner returned `action=abort, confidence=0.0` and the navigator gave up before reaching form fill. Run4 also logged a follow-up `Invalid control character at: line 11 column 42` from the same source — the model occasionally emits raw `\x00-\x1f` chars inside string values.

**How:** Two-pass JSON parse — strict first; on `JSONDecodeError`, run progressive cleanup (strip trailing commas, `//…` and `/*…*/` comments, raw control chars `\x00-\x08, \x0b, \x0c, \x0e-\x1f`) and retry. Tabs / newlines / CR are preserved (valid in JSON string values when escaped, but Kimi sometimes emits them raw — and they don't fail the parser).

**Verified (unit-level):**
```
trailing comma → action=fill_form conf=0.8
control char  → action=fill_form conf=0.8
comment       → action=fill_form conf=0.8
```
All three Kimi failure modes parse cleanly.

### Fix E (defense in depth) — `jobpulse/form_engine/field_resolver.py:best_option_match`

**Why:** Belt-and-braces. With value `"No"` (2 chars) and an options list including `"Norway"`, the prefix-match tier returns `"Norway"` because `"norway".startswith("no")` is true. If scoping ever leaks again (or a different form has both 'No' and 'Norway' as options), this prevents regression.

**How:** Short values (≤2 chars) must match as exact or whole-word prefix (`norm_opt == norm_value` or `norm_opt.startswith(norm_value + " ")`), not as a raw substring prefix.

## Outstanding caveats (non-blocking but documented)

1. **`screening_semantic_cache` always misses on this machine.** BGE-M3 (1024-dim) is unreachable; minilm (384-dim) fallback writes the wrong-dim vector and the Qdrant collection rejects the write. Every screening question therefore hits the LLM. **This was true before the session — not introduced.** It is the proximate reason the "≥50% LLM cost reduction on second run" target is missed (only 25% reduction observed). Follow-up: align embedder dim with the Qdrant collection, or run a 1024-dim local embedder.

2. **`fill_techniques.value_used` poisoning (cleaned this session).** Earlier buggy runs cached `Will you…require employment visa sponsorship…|combobox_type_to_search|Norway` in `data/form_experience.db.fill_techniques`. The technique field is what's read back, but `value_used` is diagnostic and could mislead future inspection. Cleared the bad rows (`DELETE FROM fill_techniques WHERE value_used = 'Norway'` — 2 rows removed). Since Fix C resolves the root cause, these will no longer be re-created.

3. **Three caches did not fire on this URL.** `cv_scrutiny_cache`, `portfolio_variant_cache`, `vision_classification_cache` tables are absent because their respective code paths weren't exercised on Anthropic's form (CV-too-long → B2 skipped; manual variants in DB instead of LLM bullets; DOM classifier confident enough that vision tier never engaged). All three would populate on URLs where their paths fire — they are functionally correct, just URL-conditional.
